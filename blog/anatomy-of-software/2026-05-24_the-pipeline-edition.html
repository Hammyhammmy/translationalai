Software Conduits: The Pipeline Edition

The urinary tract is a pipeline — production to expulsion, each station transforming what it receives. Your document processing pipeline has the same geometry, the same failure modes, and the same diagnostic logic.

The Premise

The Layers doc mapped your rendering stack to the abdominal wall. Layers are parallel depth — all present at once, and you localize by asking "how deep is the defect?"

But your document processing pipeline isn't layers. It's a conduit — data flows through sequential stations, each one transforming it. You localize by tracing upstream from where the output went wrong. Different geometry, different diagnostic logic.

The body has the same two patterns. The abdominal wall is layers. The urinary tract is a conduit. You wouldn't diagnose hydronephrosis by palpating the abdominal wall layer by layer. You'd trace the urinary tract upstream from the obstruction.

Each station receives the output of the previous station, transforms it, and passes it downstream. The data looks different at every boundary — raw file in, structured JSON out — but it's the same patient information, processed through a tract.

Station by Station

Station 1: Init (Kidney)

The kidney produces. Without it, nothing downstream has anything to work with.

LLMClientInitializer finds GCP credentials, validates Vertex AI enrollment, and produces an initialized LLM client. If this station fails — wrong API key, model unavailable, ZDR not enrolled — the pipeline produces nothing. Anuria.

Input Environment variables, credential files, GCP project config
→
Output instructor_client, model_name, project_id — or {success: false}
Failure Mode — Anuria

Client init fails silently on credential discovery. The health check has a 10-second timeout, but if credentials exist but are wrong (expired key, wrong project), the failure surfaces as a cryptic Vertex AI error at Station 3, not here. Same as renal failure presenting as uremia downstream — the kidney isn't producing, but the symptom shows up at the bladder.

Station 2: Validate (Ureter)

The ureter transports urine from kidney to bladder. It doesn't produce urine — it moves it, and if there's an obstruction, nothing gets through.

TextValidator combines extracted page texts, filters empties, and makes a critical gating decision: is this text processable? If combined text is under 100 characters, has no medical keywords, or matches a simple form pattern — it sets is_processable = false and the pipeline skips Station 3 entirely.

Input List of page texts (from OCR or text extraction). Could be clean. Could be garbage.
→
Output combined_text with page-break markers, is_processable flag, page count
Failure Mode — Ureteral Stricture (False Positive)

A valid medical document with sparse text (a short specialist referral, an imaging order with just a few lines) gets flagged as "not processable" because it's under 100 characters. The pipeline skips extraction entirely. The user sees "simple text processed" but doesn't know their document was never actually analyzed by the LLM. The ureter narrowed, and the document didn't pass through.

The opposite — a non-medical document that passes validation — is reflux. Garbage flows upstream into the extraction station where it doesn't belong, and the LLM produces hallucinated medical content from a grocery list.

Station 3: Extract (Bladder)

The bladder is where the real work happens — not just storage, but active processing. Detrusor contraction, compliance, capacity.

EnhancedProcessor is the most complex station. It takes combined text (or raw PDF bytes) and runs LLM extraction across 9 medical categories: demographics, allergies, medications, PMH, family history, social history, referral reason, imaging, labs. Two paths:

  • PDF-Direct: 3 batched Gemini calls on raw PDF bytes (skip OCR entirely)
  • Field Agents: 9 concurrent agents, max 2 at a time via semaphore, each making 1 LLM call with the LightProof SDK
Input Combined text with page-break markers (or PDF bytes). One string, no structure.
→
Output CategorizedStatements — 9 lists of StatementWithProvenance, each with statement, verbatim quote, page number, confidence score, integrity checks.

This is where the transformation is most dramatic. Unstructured text in, structured Pydantic models out. The data changes form entirely at this boundary — like the kidney concentrating filtrate into urine. If extraction garbles it, everything downstream faithfully processes the garbage.

Failure Mode — Bladder Dysfunction

Unanchored statements: A field agent extracts a claim but can't find a verbatim quote to anchor it. Returns {statement: "Patient has diabetes", quote: "", start: 0, end: 0, confidence: 0.0}. The statement is unanchored — the LLM hallucinated or paraphrased. But it still passes downstream into the formatter. The bladder produced, but what it produced wasn't sourced from what came in.

Partial extraction failure: One of 9 field agents times out (30-second limit, 3 retries with exponential backoff). The other 8 succeed. The summary ships with 8 of 9 categories — missing, say, imaging studies. No error shown to user. This is the graceful degradation design: better a partial summary than no summary. But the user doesn't know "imaging" is missing because the agent that extracted it failed, not because the document has no imaging.

Station 4: Format (Sphincter)

The sphincter controls release — what exits, in what form, at what time. It doesn't produce urine. It decides the final shape of what leaves.

ResponseFormatter takes the categorized statements and builds the final output: summary text, provenance structure, metadata. It filters low-confidence statements from the summary. It maps statements to page numbers. It decides whether to use the verbatim quote or the extracted statement for each field.

Input CategorizedStatements (structured Pydantic models, 9 categories with provenance)
→
Output {summary, categorized_statements, provenance, processing_metadata} — the shape that storage and the frontend expect.
Failure Mode — Sphincter Dyssynergia

The formatter filters a low-confidence statement from the summary text but keeps it in the provenance structure. The frontend shows a provenance entry with no corresponding summary line. The user sees a quote attributed to nothing — a claim the system extracted but then suppressed, without saying why. The sphincter let part of the stream through and blocked the rest, but the two outputs got out of sync. Dyssynergia — the sphincter and the detrusor aren't coordinating.

Station 5: Verify (Urinalysis)

This isn't a station in the tract itself — it's a quality check on the output. You run a urinalysis to detect problems upstream: hematuria suggests a lesion somewhere along the tract, proteinuria suggests glomerular dysfunction.

FlaggingAgent reviews the formatted summary against the provenance to catch extraction errors: high-stakes claims on thin evidence, negation flips, numeric drift (dose or lab value doesn't match the verbatim quote exactly), contradictions within the summary.

Input Summary text + provenance dict (the final formatted output)
→
Output List of VerificationFlag objects — or empty list [] if verification fails.
Failure Mode — Lab Not Resulted

The flagging agent's LLM call fails (timeout, rate limit). It returns an empty list. The summary ships without verification — no flags, no warnings, no indication that verification was even attempted. The urinalysis didn't fail visibly — it just never resulted. The clinician doesn't know the test wasn't run. In your app: the user doesn't know the safety-critical flagging step was skipped. The summary looks clean because nobody checked it, not because it passed.

Station 6: Store (Voiding)

The final act. Urine exits the body. The processed summary exits the pipeline and enters storage — GCS blob or Postgres row — where it becomes the source of truth that the layer stack will display.

Input Final result dict (summary + provenance + flags + metadata)
→
Output Timestamped JSON at {user_id}/{filename}_{timestamp}.json in GCS or local filesystem.
Failure Mode — Outlet Obstruction

Storage write fails — GCS timeout, disk full, permission error. The entire pipeline ran successfully. The summary is perfect. But it's lost. The temp file is cleaned up in the finally block. The job status was never updated from "processing." The user sees an eternal spinner. Retention — the data was produced, processed, formatted, and verified, but never voided. And there's no catheter (no retry, no fallback storage).

Conduit vs. Layer: The Key Difference

In the layer stack, all layers are present simultaneously and the same data passes through all of them. A stale cache at layer 4 presents as wrong data at layer 1. You localize by depth.

In a conduit, each station transforms the data. The output of Station 3 is structurally different from its input — unstructured text becomes Pydantic models. You can't just "check each layer" because the data at each boundary is a different shape. You have to check input vs. output at each station.

The Diagnostic Move

Layer diagnosis: "Is the data correct at this depth?"

Conduit diagnosis: "Did this station produce correct output from correct input?" — Two checks per station, not one. If the input was good and the output was bad, the station is the lesion. If the input was already bad, keep tracing upstream.

The Diagnostic Algorithm

A malformed summary is on screen. The user sees wrong medications, missing allergies, a garbled referral reason. Where do you look?

Don't start at Station 1. Start at the output and trace upstream.

Step 1 — Check the Voided Output (Storage)

Question: Is the stored JSON correct? Or is the display wrong?

# Read the raw stored output:
gsutil cat gs://chartprepcanada/users/USER_ID/filename_timestamp.json | python -m json.tool

# Or locally:
cat gcs_local_version/USER_ID/filename_timestamp.json | python -m json.tool

# Check: does the JSON match what's on screen?
# If JSON is correct but screen is wrong → layer problem (cache/rendering), not pipeline
# If JSON is wrong → pipeline problem. Keep tracing upstream.

If stored output is correct → the pipeline is fine. The problem is in the layer stack (cache, state, rendering). Switch to layer diagnosis.
If stored output is wrong → pipeline defect. Go to Step 2.

Step 2 — Check the Flags (Urinalysis)

Question: Did the flagging agent catch the problem? Did it even run?

# In the stored JSON, check verification_flags:
cat output.json | python -c "
import json, sys
data = json.load(sys.stdin)
flags = data.get('verification_flags', [])
print(f'Flags found: {len(flags)}')
for f in flags:
    print(f'  - {f.get(\"type\")}: {f.get(\"message\")}')"

# If flags is empty [] → was it verified and clean, or did verification fail?
# Check server logs for: "Flagging agent failed" or "verification skipped"

If flags caught the issue → flagging works, but the summary shipped anyway (by design — flags are advisory).
If flags are empty → either genuinely clean, or flagging silently failed. Check logs. Either way, the defect is upstream.

Step 3 — Check the Formatted Output (Sphincter)

Question: Did the formatter receive correct categorized statements and produce wrong output?

# In the stored JSON, compare categorized_statements to final_summary:
cat output.json | python -c "
import json, sys
data = json.load(sys.stdin)
cats = data.get('categorized_statements', {})
for category, statements in cats.items():
    print(f'{category}: {len(statements)} statements')
    for s in statements[:2]:
        print(f'  statement: {s.get(\"statement\", \"\")[:80]}')
        print(f'  quote:     {s.get(\"quote\", \"\")[:80]}')
        print(f'  confidence: {s.get(\"confidence\", \"?\")}')"

# Compare: are the categorized statements correct but the summary wrong?
# Check for summary/provenance mismatch (statement in provenance, missing from summary)

If categorized statements are correct but summary is wrong → formatting bug. The sphincter garbled the output. Fix at ResponseFormatter.
If categorized statements are already wrong → extraction defect. Go to Step 4.

Step 4 — Check the Extraction (Bladder)

Question: Did the LLM extract correct statements from the combined text?

# Check for unanchored statements (no verbatim quote):
cat output.json | python -c "
import json, sys
data = json.load(sys.stdin)
for cat, stmts in data.get('categorized_statements', {}).items():
    unanchored = [s for s in stmts if not s.get('quote')]
    if unanchored:
        print(f'{cat}: {len(unanchored)} UNANCHORED statements')
        for s in unanchored:
            print(f'  → {s[\"statement\"][:80]}')"

# Check integrity scores:
cat output.json | python -c "
import json, sys
data = json.load(sys.stdin)
for cat, stmts in data.get('categorized_statements', {}).items():
    low = [s for s in stmts if s.get('confidence', 1) < 0.5]
    if low:
        print(f'{cat}: {len(low)} LOW CONFIDENCE')
        for s in low:
            print(f'  → {s[\"statement\"][:60]} (conf: {s[\"confidence\"]})')"

# If a specific field agent failed, check server logs:
# grep "field_agent.*failed\|field_agent.*timeout\|retry.*attempt" server.log

If extraction produced unanchored/wrong statements from correct text → LLM extraction bug. Prompt engineering, model issue, or Pydantic schema mismatch.
If the combined text fed to extraction was already wrong → validation/OCR defect. Go to Step 5.

Step 5 — Check the Validated Text (Ureter)

Question: Was the text correctly combined? Was the processability decision right?

# Check processing_metadata for the validation decision:
cat output.json | python -c "
import json, sys
data = json.load(sys.stdin)
meta = data.get('processing_metadata', {})
print(f'Method: {meta.get(\"processing_method\")}')
print(f'Enhanced skipped: {meta.get(\"enhanced_processing_skipped\", False)}')
print(f'Pages: {meta.get(\"pages_processed\", \"?\")}')"

# If enhanced_processing_skipped is True:
#   → TextValidator decided the text wasn't medical or was too short
#   → Was that decision correct? Check the original document.
#   → If the document IS medical → stricture: validation threshold too aggressive

# Check combined_text length and content:
# Add temporary logging to TextValidator.validate_and_combine_pages():
#   logger.info(f"Combined text length: {len(combined_text)}")
#   logger.info(f"First 200 chars: {combined_text[:200]}")

If text was incorrectly classified as non-processable → validation stricture. Fix the threshold in TextValidator.
If combined text is garbled or empty → OCR/extraction defect. Go to Step 6.

Step 6 — Check the Source (Kidney)

Question: Was the original file readable? Did OCR/text extraction work?

# Re-process the same file and check raw text extraction:
# Look at the upload route logs for the original processing:
# grep "process_single_file\|text_extraction\|ocr" server.log

# For PDF-direct path: the issue might be that PDF bytes were used
# but the PDF is a scanned image (no embedded text).
# Check processing_method in metadata — "pdf_direct" vs "field_agents"

# If PDF-direct was used on a scanned PDF → silent failure
# The LLM received image pages but no selectable text
# Output depends entirely on Gemini's vision capability

# For OCR path: check if the temp file was readable:
# Was the upload corrupted? Was the file format supported?

If the source file is unreadable → upstream of your pipeline entirely. Bad scan, corrupted upload, unsupported format.
If the file is readable but text extraction failed → client init or OCR defect. The kidney didn't produce.

You stop at the first station where correct input produced incorrect output. That's the lesion.

The Shortcut — Same as Layer Diagnosis

The user's complaint tells you where to start. "The summary is missing allergies" → jump straight to Station 3, check the allergies field agent. "It says 'simple text processed'" → Station 2 made the wrong gating decision. "I uploaded a PDF and got nothing" → Station 1, check if the client initialized. Don't start at the output every time. Let the symptom tell you the station.

Surgical Principles for Conduits

Layer principles are about protecting depth boundaries. Conduit principles are about protecting station boundaries — making sure each station's input is clean, its output is correct, and failures don't propagate silently.

StationPrincipleAnatomy Parallel
Init Fail loud at the source. If the kidney doesn't work, everything downstream is pointless. Don't let the pipeline proceed with a broken client — surface the failure immediately. Renal agenesis is diagnosed at birth, not when the patient presents with uremia at 40.
Validate Gate explicitly. If you skip a document, tell the user why. Silent skips are strictures — the data didn't pass, and nobody knows. Ureteral stents are visible on imaging. Your validation gate should be visible in the UI.
Extract Anchor every claim. An unanchored statement (no verbatim quote) is an unverified claim from the LLM. Mark it, flag it, or drop it — but don't let it pass as sourced. Urine should come from filtered blood, not from somewhere else. If you can't trace the provenance, don't trust the output.
Format Keep summary and provenance in sync. If you filter a statement from the summary, filter it from provenance too. Dyssynergia between the two outputs confuses the user. Detrusor-sphincter coordination. Both outputs must agree on what passes through.
Verify If verification didn't run, say so. An empty flag list must distinguish "verified clean" from "verification failed." Silent failure here is a lab that never resulted. A missing urinalysis result is not a normal urinalysis.
Store Write atomically or not at all. If storage fails, the pipeline result must be recoverable — retry, fallback path, or at minimum a clear error status. Don't clean up the temp file before confirming the write. Post-void residual check. Confirm the bladder actually emptied before you close.

Where Conduit Meets Layers

The pipeline doesn't exist in isolation. It terminates by feeding into the layer stack. Station 6 (storage) writes the JSON that becomes the source of truth at Layer 6 (server/database) in the layer model.

This means a single bug can span both patterns. A malformed summary on screen could be:

  • Pipeline defect: The extraction station garbled the data. Every layer displays the garbage faithfully. Fix at the station.
  • Layer defect: The pipeline produced correct data, but the cache is serving a stale version. Fix at the layer.
  • Boundary defect: The pipeline stored correctly, but the layer stack loaded it wrong (deserialization bug, wrong JSON key). Fix at the boundary where conduit meets layers.

The first diagnostic question is always: which pattern am I in? Check the stored JSON. If it's correct, you're in the layer stack. If it's wrong, you're in the pipeline. Don't mix the diagnostic algorithms.

The Urinary Tract to Abdominal Wall Junction

The ureter passes through the abdominal wall to reach the bladder. The pipeline passes through the layer stack to reach the screen. They're different systems with different geometries, but they share a boundary. Pathology at that boundary — a ureteral injury during abdominal surgery, a deserialization bug at the storage-to-cache handoff — requires you to think in both patterns simultaneously. Know which system you're diagnosing, and know where they meet.
