Software Networks: The Vasculature Edition

When one artery occludes, the symptom isn't at the artery — it's across the entire territory it supplies. When your Firebase token expires, the symptom isn't in one endpoint — it's across every authenticated consumer simultaneously. Same pattern, same diagnosis.

The Premise

The Layers doc taught depth localization — go deeper until you find the boundary. The Pipeline doc taught upstream tracing — follow the flow back to the station that corrupted it.

Neither works when the symptom is "everything broke at once."

When a user reports that uploads fail, the sidebar won't load, and the file list is empty — all at the same time — that's not a layer problem (wrong depth) or a pipeline problem (wrong station). It's a network problem. One trunk went down, and every branch in its territory lost supply. You diagnose it by mapping the territory, not by going deeper or tracing upstream.

The body does this with vasculature. Software does it with auth, events, and shared state.

Your Auth Vasculature

Firebase auth in ChartPrepper is an arterial tree. One trunk — the ID token — branches to supply every authenticated consumer in the app.

Firebase ID Token (trunk)
┌──────────┼──────────┐
HTMX Requests htmx:configRequest injects Authorization header on every request
fetch() Calls fetch-middleware.js adds Bearer token via waitForToken()
Cookie Sync POST /verify-firebase-token sets HttpOnly cookie as fallback
File Operations Upload, delete, hide/unhide — all require user_id from token
LightScribe Mount /ls/* routes use same auth middleware

When the token is valid, all branches receive supply. When it expires or gets revoked, every branch loses supply simultaneously. This simultaneity is the hallmark of a network pattern — and the diagnostic key.

Token Expiry: An Ischemic Event

Let's trace a real failure through the network — your Firebase token expires mid-session.

The timeline

  1. T+0: Token was issued 55 minutes ago. TokenManager has it cached. Everything works.
  2. T+5min: Proactive refresh loop (runs every 5 minutes) checks expiration. Token expires in <5 minutes. Calls user.getIdToken(true). Fresh token obtained. All branches re-supplied. User sees nothing.

That's the healthy path — collateral circulation maintained the supply. But what if the refresh fails?

  1. T+0: Token expires. Refresh loop fires. user.getIdToken(true) fails (network down, Firebase outage, session revoked server-side).
  2. T+0: clearInterval() — the refresh loop stops retrying. No exponential backoff. No retry count. The pump just stops.
  3. T+1: User clicks "Upload." htmx:configRequest fires. Calls TokenManager.getToken(). Token is expired. Calls user.getIdToken(true) — fails again. Request goes out without Authorization header.
  4. T+1: Server receives unauthenticated request. get_user_id_from_token() raises HTTPException(401). Global exception handler converts to 200 + X-Auth-Error: token_expired header.
  5. T+1: cold-start-detector.js sees X-Auth-Error → shows "Session ended" banner.
  6. T+2: User clicks sidebar. Same cascade. Another 401-as-200. Another banner.
  7. T+2: htmx:responseError fires. Attempts one more user.getIdToken(true). Fails. Clears all tokens (localStorage, cookie, cache). Shows "Session Expired" modal. Redirects to /login.
The Ischemic Cascade

Every authenticated endpoint failed simultaneously. Not one by one, not in sequence — all at once. The trunk occluded, and the entire territory went ischemic. The symptom was "everything is broken." The lesion was one token.

If you diagnosed this as a layer problem ("the sidebar isn't rendering correctly") or a pipeline problem ("the upload flow is broken"), you'd investigate those specific systems and find nothing wrong with them. They're structurally intact. They're just not receiving supply.

Territory Mapping: The Diagnostic Method

In vascular pathology, you don't trace one vessel. You map the territory — what's affected and what's spared. The boundary between affected and spared tells you the branch point.

Software networks work the same way. When multiple things break, don't investigate each one separately. Map what's affected vs. what's spared.

Example: "Uploads fail and the file list is empty"

Affected
Upload (POST /api/upload) — 401
File list (GET /api/files) — 401
Delete file — 401
Export — 401
Billing page — 401
Spared
Login page loads — no auth required
Static assets (CSS, JS, icons) — no auth
Firebase config endpoint — exempt
Health check — exempt
App shell renders — client-side only

Interpretation: Everything that requires get_user_id_from_token() is affected. Everything exempt is spared. The boundary is the auth middleware. The trunk is the Firebase token. This is a token-level occlusion, not a bug in uploads, files, or billing individually.

Example: "The file list is stale but uploads work"

Affected
Sidebar file list — stale
Calendar view — stale
File count badge — wrong number
Search results — searching stale data
Spared
Upload succeeds (server confirms)
Summary content loads correctly when clicked
API returns correct data in Network tab
New file appears after manual refresh

Interpretation: Everything bound to Alpine.store('files') is stale. Everything that reads directly from the server is fine. The boundary isn't auth — it's the Alpine store. This is an MCA stroke, not an aortic dissection. The trunk is Alpine.store('files'), and the territory is every component watching that store key.

This is not a layer problem (the cache might be fine — the store just wasn't updated after upload). And it's not a pipeline problem (the upload pipeline succeeded). It's a network problem — one store key didn't propagate, and its entire territory went stale.

The Diagnostic Principle

Don't trace one path. Compare territories. The boundary between affected and spared tells you the branch point. If you investigate each broken feature independently, you'll find five "bugs." If you map the territory, you'll find one trunk.

The Three Networks in Your App

1. Auth Network (Arterial — supplies access)

Firebase SDK (aorta)
┌──────┼──────┐
Token Generation user.getIdToken()
Proactive Refresh 5-min loop in firebase_auth.js
On-demand Refresh htmx:configRequest handler

┌──────────────┼──────────────┐
Header Path Authorization: Bearer → all HTMX/fetch routes
Cookie Path HttpOnly id_token cookie → fallback for page loads

2. Store Network (Nervous — propagates state)

Alpine stores are like the nervous system — one signal source propagates to many effectors. A store key update triggers every x-text, x-for, x-show, and x-effect that watches it.

Store Key (Nerve Trunk)Territory (Effectors)Failure Mode
Alpine.store('files') Sidebar file list, calendar view, badge counts, search filter, hidden files pill Store not updated after upload → entire file UI is stale. Motor neuron lesion — the signal isn't reaching any of the muscles.
Alpine.store('sidebar') Sidebar visibility, width, resize handle, mobile overlay Store says "open" but CSS says "hidden" → phantom panel. Upper vs. lower motor neuron — the signal exists but the effector isn't responding.
fileTableManager Sort order, view mode, selection state, hidden file tracking Manager state desynchronized from Alpine store → two sources of truth. Conflicting innervation — two nerves trying to control the same muscle.

3. Event Network (Lymphatic — propagates signals)

Custom events and HTMX lifecycle events propagate through the DOM like lymphatic drainage — the event originates at a source and flows through a chain of listeners.

Event (Drainage Basin)Listeners (Nodes)Failure Mode
auth-ready file-table-manager (initial load), app.js (email display), app-help.js (help panel) If auth-ready never fires (Firebase SDK fails to load), all three listeners wait forever. The app looks loaded but nothing initializes. Lymphedema — fluid accumulates because drainage never started.
chartprep:files-changed debug_overlay (stats), tab_switching (badge counts) If this event fires with wrong data, both listeners update incorrectly. The event is the sentinel node — if it's wrong, the entire downstream basin is wrong.
htmx:afterSwap offline-cache sync, Alpine re-initialization on swapped content If a swap silently fails, the cache doesn't sync and Alpine doesn't reinitialize. The event didn't fire, so downstream listeners don't know anything happened. Sentinel node negative — no drainage, but not because there's nothing to drain.

Collateral Circulation

The circle of Willis provides collateral flow — if one carotid is occluded, the contralateral carotid and vertebrobasilar system can partially supply the affected territory through communicating arteries. The infarct is smaller because collaterals limit the damage.

Your app has collateral circulation built in:

OcclusionCollateralCoverage
Authorization header fails HttpOnly cookie fallback Partial — cookie only works on full page loads, not HTMX requests. Covers page navigation but not in-page API calls.
Token expired on request On-demand refresh in htmx:configRequest Full — silent refresh before the request fires. User never sees the expiry. This is the strongest collateral.
Vertex AI (Gemini) unavailable LLM_PROVIDER env var → OpenAI, Bedrock, Anthropic Full (if configured) — but requires restart with new env var. Not automatic. Manual cross-clamping.
Server returns 503 (cold start) cold-start-detector.js shows banner, user refreshes Manual — the "collateral" is the user pressing F5. Server warms up on next request. Temporary ischemia, not infarction.
GCS unavailable localStorage cache + buildSummaryHTML() Read-only — cached summaries display from localStorage. Writes fail. The app is read-only during the outage. Like perfusing just enough to prevent necrosis but not enough for function.
Collaterals Don't Fix the Occlusion

The circle of Willis doesn't re-open the occluded carotid. It limits the infarct while you address the root cause. Similarly, the HttpOnly cookie fallback doesn't fix the expired token. The localStorage cache doesn't fix the GCS outage. Collaterals buy time. They don't fix trunks. If you see a collateral engaged (cookie being used instead of header, cache serving instead of server), that's a diagnostic sign — a trunk is down somewhere.

The Diagnostic Algorithm

Multiple features broken at once. Here's the procedure:

Step 1 — Confirm It's a Network Problem

Question: Are multiple unrelated features broken simultaneously?

// Quick check: is this a network problem or coincidence?
// Try 3 unrelated actions:
//   1. Click a file in sidebar (needs auth + store)
//   2. Try uploading (needs auth + pipeline)
//   3. Check if login page loads (no auth needed)

// If all three fail → trunk occlusion (auth or server)
// If 1 and 2 fail but 3 works → auth trunk
// If only 1 fails → not a network problem, go to layer/pipeline diagnosis
Step 2 — Map the Territory

Question: What's affected and what's spared?

// Check auth state:
firebase.auth().currentUser                    // null? → Firebase SDK issue (aorta)
await firebase.auth().currentUser.getIdToken() // throws? → token refresh failed
localStorage.getItem('id_token')               // null? → token never cached

// Check which requests fail:
// DevTools → Network tab → filter by status code
// All 401s? → auth trunk
// Mix of 401 and 200? → some routes exempt (check exempt_paths in main.py)
// All 503? → server trunk (Uvicorn/cold start)

// Check store state:
Alpine.store('files')   // populated or empty?
Alpine.store('sidebar') // correct state?

// If stores are empty but API returns data → store network (nervous system)
// If API returns 401 → auth network (arterial)
// If API returns 503 → server network (cardiac)
Step 3 — Identify the Branch Point

Question: Where does the boundary between affected and spared fall?

// Auth network — check the trunk chain:
firebase.auth().currentUser           // Firebase SDK loaded? (aorta)
  → .getIdToken()                     // Token available? (common carotid)
    → TokenManager.getToken()         // Cache valid? (internal carotid)
      → htmx:configRequest injects    // Header path? (MCA)
      → cookie exists?                // Cookie path? (ACA — collateral)

// Store network — check the signal chain:
// Did the data source fire an update?
Alpine.store('files')                 // Is the store object correct?
document.querySelector('[x-for]')     // Are bindings present?
// If store is correct but UI is wrong → binding issue (effector)
// If store is stale → who was supposed to update it? (signal source)

// Event network — check event propagation:
document.addEventListener('auth-ready', e => console.log('auth-ready fired', e.detail));
document.addEventListener('chartprep:files-changed', e => console.log('files-changed', e.detail));
// Reload page. Which events fire? Which don't?
// Missing event → the source never dispatched. Find the source.
Step 4 — Check Collaterals

Question: Are fallback paths engaged? That tells you the primary trunk is down.

// Is the cookie being used instead of the header?
// Check server logs for: "Using token from cookie" vs "Using token from header"
// If cookie → header path is down. Check htmx:configRequest handler.

// Is localStorage serving cached content?
// Check if offline-cache.js is serving summaries from cache:
console.log('Cache keys:', Object.keys(localStorage).filter(k => k.includes('summary')));
// If summaries load from cache but API calls fail → GCS is down, cache is collateral

// Did cold-start-detector fire?
// Check for the red banner. If present → server trunk was temporarily down.

// Is the proactive refresh loop still running?
// Add this to console to check:
// If firebase_auth.js clearInterval() was called, the refresh loop stopped.
// The "pump" (proactive refresh) stopped. Only on-demand refresh remains.

The branch point is where affected meets spared. Fix the trunk, and the entire territory recovers.

The Network Shortcut

"Logging out and back in fixes everything" → auth trunk. The re-login obtained a fresh token, re-supplying all branches. "Refreshing the page fixes it" → could be auth (cookie collateral kicks in) or store (Alpine re-initializes from server). "It fixes itself after a minute" → the proactive refresh loop re-supplied the trunk. Transient ischemia, not infarction.

Surgical Principles for Networks

PrincipleSoftwareAnatomy
Maintain collateral circulation. Every trunk should have a fallback path. Authorization header → cookie. Server data → localStorage cache. Primary LLM → fallback provider. Don't assume the primary path is always available. The circle of Willis exists because the brain can't tolerate ischemia. Your auth system can't either.
Monitor the trunk, not the branches. Check token validity centrally (TokenManager), not at each consumer. If you instrument each API route's auth separately, you'll see five failures instead of one cause. You monitor carotid flow with a Doppler, not by checking perfusion in each cerebral territory independently.
Fail loud at the trunk. When the refresh loop stops (clearInterval()), it stops silently. No log, no warning, no retry. The trunk occluded and nobody was notified. Log trunk failures at ERROR level. A silent trunk death guarantees a cascade. Carotid stenosis is screened for because a silent occlusion becomes a catastrophic stroke.
Map territories in advance. Before a bug, know which consumers depend on which trunk. Document that Alpine.store('files') supplies the sidebar, calendar, badges, and search. When the store breaks, you already know the blast radius. You know the MCA territory (face, arm, speech) before you see the stroke. You don't figure it out during the code.
Don't create single points of failure. The proactive refresh loop has no retry, no fallback, no notification. If it fails, the only remaining path is on-demand refresh in htmx:configRequest. One failure and you're down to your last collateral. End-artery anatomy (retinal artery, coronary) means no collateral. Occlusion = infarction. Design redundancy into your trunks.

Where Networks Meet Layers and Conduits

A token expiry (network) causes an upload failure (pipeline) that the user sees as a broken screen (layers). Three patterns, one incident. The diagnostic triage is:

  1. Is it simultaneous and multi-feature? → Network. Map the territory.
  2. Is it one feature with corrupted output? → Pipeline. Trace upstream.
  3. Is it one feature with wrong display? → Layers. Go deeper.

Get the pattern right first. The diagnostic algorithm follows from the geometry.

The Acute Abdomen — All Three at Once

A mesenteric artery occlusion (network) causes bowel ischemia that corrupts the GI pipeline (conduit), which leads to perforation through the peritoneum (layers). In your app: Firebase SDK fails to load (network — aortic level), which causes an upload to fail mid-processing (conduit — extraction station gets 401 from Vertex AI because credentials share the GCS service account), which writes a partial result to storage that the layer stack faithfully displays as a garbled summary (layers). Three patterns, one root cause, one fix: restore the trunk.
