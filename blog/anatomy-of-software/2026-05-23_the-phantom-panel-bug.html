The Phantom Panel

A Layer 2 bug that masqueraded as a Layer 5 performance problem, a Layer 1 rendering glitch, and a Layer 3 state issue — all at the same time. And why it took us hours to find a one-line fix.

The Symptom

On mobile Safari, open the calendar view. Tap a file to read its summary. Tap back. The calendar view is gone. The sidebar looks broken — wrong layout, wrong spacing, like it forgot what it was doing. Tap on the Notes tab and back to Documents: everything is fine again. Refresh the page: fine again.

It only happens on mobile. It only happens after viewing a file. And it fixes itself if you switch tabs.

The Misleading Clue

The calendar also felt slow to load. The server logs showed get_summaries: returning 401 documents on every file click. So the first instinct was: this is a performance problem at Layer 5/6. We spent significant time optimizing the SQL query, reducing JSON payload sizes, and adding client-side caching. All good improvements — but none of them fixed the actual bug.

Why It Took So Long

This bug was hard to find for a specific, instructive reason: the symptom shifted depending on which layer you were looking at. It was a chameleon — it looked like a different bug at every layer of investigation.

1. It looked like a performance problem (Layer 5/6)

The server logs were alarming. Every time you tapped a file, the summary endpoint loaded all 401 documents from Postgres just to return one. The calendar endpoint fetched full JSON blobs when it only needed four fields. These were real problems — genuinely wasteful queries that needed fixing.

But they were independent problems, not the cause of the broken layout. We fixed them (summary route now does a single-row query; calendar extracts only needed fields via data->>'field_name'; client caches for 30 seconds). The calendar loaded faster. The sidebar was still broken.

The Trap

When you find a real problem at one layer, it's psychologically very hard to keep looking at other layers. You've found something wrong, you've fixed it, you expect the symptom to resolve. When it doesn't, you look for more problems at the same layer instead of stepping back and re-examining the original symptom. This is anchoring bias — the same cognitive trap that makes premature closure dangerous in clinical diagnosis.

2. It looked like an iOS rendering bug (Layer 1)

The calendar panel had correct data in the Alpine store. The DOM elements existed. But they weren't visible. On iOS Safari, this pattern — "DOM is right but nothing renders" — is a known class of bugs where WebKit doesn't recalculate layout after a display property change. We'd already built a workaround for this: the translateZ(0) repaint hack in switchSidebarTab().

But that repaint hack only targeted the documents and notes panels. The calendar panel was never included. So we added it. Still broken — because the repaint hack only fires during switchSidebarTab(), and the broken flow never calls switchSidebarTab().

3. It looked like an Alpine state problem (Layer 3)

Earlier in the session we'd migrated the calendar from vanilla JS rendering to Alpine.js templates. This was fresh surgery. When the calendar didn't render correctly after tapping back, the natural suspicion was: "did the migration break something? Is Alpine not reacting to the state change?"

We checked. Alpine's state was correct: viewMode === 'calendar', calData was populated, calGridCells returned the right cells. The state was perfect. Alpine was doing its job. The problem was below Alpine — at the DOM manipulation layer that Alpine doesn't control.

The Actual Bug

Two functions. One sets a CSS property. The other doesn't unset it.

handleFileClick() — entering summary view on mobile

// On mobile: switch to summary view
if (window.innerWidth <= 768) {
    document.body.classList.add('mobile-summary-view');
    const mainWrapper = document.getElementById('main-content-wrapper');
    if (mainWrapper) mainWrapper.style.display = 'flex';  // Sets inline style
}

mobileGoBack() — returning to the file list

function mobileGoBack() {
    document.body.classList.remove('mobile-summary-view');
    // That's it. That's the whole function.
}

The CSS class mobile-summary-view hides the sidebar and shows the main content. Removing it should reverse those effects. But handleFileClick also set an inline style on #main-content-wrapper: style="display: flex".

Inline styles override CSS rules. The mobile CSS says #main-content-wrapper { display: none; }, but the inline display: flex trumps it. After mobileGoBack(), the main content wrapper was still display: flex — invisible content taking up layout space, pushing the sidebar into a broken state.

The Lesion

mobileGoBack() reversed the CSS class but not the inline style. One property, one element, one missing line of code. Everything else — the slow queries, the missing repaint hack, the Alpine migration — was either a separate problem or a consequence of this one.

The Fix

function mobileGoBack() {
    document.body.classList.remove('mobile-summary-view');
    var mainWrapper = document.getElementById('main-content-wrapper');
    if (mainWrapper) mainWrapper.style.display = 'none';     // Reset the inline style
    // ... plus _syncViewPanels() and repaint hack
}

One line: mainWrapper.style.display = 'none'. That's the fix for the layout bug. Everything else we added (the repaint hack for the calendar panel, the _syncViewPanels() call) is defensive hardening — good practice, but not the root cause.

Why switchSidebarTab Fixed It

The user noticed that clicking Notes and then back to Documents "reset" the broken state. This was the critical clue, and here's exactly why it worked:

// Inside switchSidebarTab():
if (isMobile && mainWrapper) {
    mainWrapper.style.display = isFullScreen ? 'flex' : 'none';  // Resets the inline style!
}

switchSidebarTab('notes') sets mainWrapper.style.display = 'none' as a side effect. It cleans up the stale inline style that mobileGoBack() left behind. The bug silently fixes itself — which is why it looked intermittent and made us doubt our diagnosis at every turn.

The Self-Healing Pattern

Bugs that fix themselves through unrelated actions are the hardest to diagnose. In medicine, it's the equivalent of a symptom that resolves with positional change — "it only hurts when I sit, and standing fixes it." You start investigating the standing mechanism instead of the sitting mechanism. The fix is a red herring that pulls you away from the cause.

The Lesson

Inline styles are fascial adhesions

An inline style="display: flex" overrides CSS class rules the same way a fascial adhesion overrides the normal glide plane between layers. The CSS class removal (mobile-summary-view) should have been enough to restore the layout — just as removing superficial tension should let the layers underneath return to their normal position. But the adhesion (inline style) anchored the element in the wrong state, and no amount of work at other layers could fix it.

Noisy logs are incidental findings

The 401-document query was a real problem, like an incidental finding on imaging. It deserved attention and treatment. But it wasn't the chief complaint. We got pulled into treating the incidentaloma (performance) when the patient presented with a functional complaint (layout breaks on back-navigation). In retrospect, the performance issue had been present since the app was built — it wasn't new. The layout bug was new. New symptoms need new explanations.

The user's workaround IS the diagnosis

"Clicking Notes and back fixes it" is not just a workaround — it's a precise diagnostic statement. It tells you: (1) the bug is in the back-navigation path specifically, not in the calendar itself, (2) the tab-switching path has something the back-navigation path lacks, (3) the fix is at the point where those two paths diverge. We should have diff'd those two code paths the moment we heard this. Instead, we spent another hour on query optimization.

The Rule

When the user tells you the exact maneuver that fixes a bug, stop everything and compare that maneuver's code path with the broken code path. The difference between them is the bug. Every minute spent elsewhere is wasted.
