Three Patterns of the Body

Every anatomical structure is one of three geometric patterns. Every software system is too. The pattern tells you how to diagnose it.

The Claim

You've spent years learning thousands of structures. But they all organize into three geometric patterns — and those three patterns are the same ones that organize every software system you'll ever debug.

This isn't metaphor. The body and software are both engineered systems with the same constraints: they need to protect (layers), transport (conduits), and coordinate (networks). The geometry follows from the function.

Once you see the pattern, you know the diagnostic logic before you know the specific anatomy.

Pattern 1 — Layers

Parallel depth. Localize by going deeper.

Body: Abdominal wall (skin → subcutaneous → muscle → fascia → peritoneum). Scalp (SCALP mnemonic). Meninges (dura → arachnoid → pia). Vessel walls (intima → media → adventitia). Every boundary you cross changes the tissue type, the function, and the failure mode.

Software: The rendering stack. A request passes through all layers simultaneously — screen, reactivity, state, cache, network, server. Each layer transforms the same data. A defect at any depth presents at the surface.

Skin / Rendering (Tailwind, HTML)
Subcutaneous / Reactivity (Alpine, HTMX swap)
Muscle / State (Alpine store)
Fascia / Cache (localStorage)
Preperitoneal / Network (HTMX fetch, fetch())
Peritoneum / Server (Uvicorn → FastAPI → Postgres)

Diagnostic logic: Start at the surface. Go one layer deeper at each step. Stop at the first boundary where correct becomes incorrect. That's the lesion.

Signature failure: Referred symptoms. The defect is at layer 4 but the symptom is at layer 1. You blame what you can see (Alpine) instead of what you can't (cache). Same as a transversalis fascia defect presenting as a skin-level bulge.

Pattern 2 — Conduits

Serial stations. Localize by tracing upstream.

Body: Urinary tract (kidney → ureter → bladder → sphincter → urethra). GI tract (mouth → esophagus → stomach → small bowel → colon → rectum). Biliary tree (hepatocyte → canaliculus → bile duct → gallbladder → common bile duct → duodenum). Each station has a specific job — production, transport, storage, control, expulsion. The order matters. Flow is one-directional.

Diagnostic logic: Start at the output (what came out wrong). Trace upstream station by station. Check the output of each station. The first station that produced bad output from good input — that's the lesion.

Signature failure: Obstruction. Something blocks or corrupts the flow at a station, and everything downstream is affected. Hydronephrosis = ureter obstruction. A malformed summary = the LLM extraction step garbled the data, and formatting faithfully formatted the garbage. The formatter isn't broken. The extraction is.

Key Difference from Layers

In a layer system, all layers are present simultaneously and the same data passes through all of them. In a conduit, each station transforms the data — the output of one station is different from its input. That's why you can't just "check each layer" — you have to check the input and output at each station to find where the transformation went wrong.

The GI parallel

Your CI/CD pipeline is a GI tract. Code enters (ingestion), gets tested (digestion — breaking it down to see if it's viable), gets built into an artifact (absorption — extracting the useful form), and gets deployed (elimination — pushing it out into the world). A failed test doesn't mean the deployment system is broken. It means digestion caught something indigestible. That's working correctly.

The biliary parallel

Your logging pipeline — from application event to structured log to dashboard — is a biliary tree. The liver produces bile (your app produces log events). The bile ducts transport it (your logging library serializes and routes it). The gallbladder concentrates and stores it (your log aggregator indexes and retains it). Cholestasis — backup from an obstructed duct — looks like your logging pipeline when the aggregator goes down. Events are produced but never reach the dashboard. The app is fine. The transport is blocked.

Pattern 3 — Networks

Branching distribution. Localize by mapping the territory.

Body: Arterial vasculature (aorta branches to organs, each with its own territory). Lymphatic drainage (nodal basins, sentinel nodes). Nervous system (motor cortex → corticospinal tract → anterior horn → peripheral nerve → muscle). Venous return. Dermatomes. Bronchial tree. One trunk fans out to many endpoints. The endpoints define territories.

Software: Authentication, event systems, configuration propagation. One source fans out to many consumers. A failure at the trunk affects all downstream territories simultaneously.

Diagnostic logic: Don't trace a single path — map the distribution of the deficit. What's affected and what's spared? The boundary between affected and spared territories tells you the branch point. If API routes fail but WebSocket works, the lesion is in the API middleware, not the token itself. If everything fails, the lesion is at the trunk.

Signature failure: Ischemia. A branch goes down and its entire territory loses supply. Token expiry kills every authenticated endpoint at once — not one at a time, not in sequence, all simultaneously. That simultaneity is the hallmark of a network pattern failure. Layer failures present at the surface. Conduit failures present downstream. Network failures present in a territory.

Collateral Circulation = Fallback Systems

The circle of Willis provides collateral flow — if one carotid is occluded, the other side can partially supply the territory through communicating arteries. Software equivalents: retry logic, fallback auth (cookie if header fails), CDN serving cached static files when the origin is down, your LLM_PROVIDER env var that can switch from Vertex AI to OpenAI. Collaterals don't fix the occlusion. They limit the infarct size while you fix the trunk.

The First Question

Before you diagnose, before you investigate, before you touch the code — ask one question:

What pattern am I looking at?

The answer determines your entire diagnostic approach.

Layers
"Which depth?"
Start at the surface. Go one layer deeper at each step. Stop at the boundary where correct becomes incorrect.
Conduit
"Which station?"
Start at the output. Trace upstream. Check input vs. output at each station. The first bad transformation is the lesion.
Network
"Which territory?"
Map what's affected vs. spared. The boundary between them tells you the branch point. Don't trace one path — compare territories.

Getting the pattern wrong costs you the most time. If you apply layer thinking to a pipeline bug, you'll examine each "depth" of the rendering stack and find nothing wrong — because the data is correct at every layer. It was just corrupted at a pipeline station before it entered the layer stack. You'd be examining the abdominal wall for a ureteral stone.

How to Tell Which Pattern You're In

Triage Heuristics
It's a LAYER problem if:
  → The data is on screen but wrong or missing
  → "Refresh fixes it" or "it looks right in the API response"
  → The symptom is visual — wrong content, wrong layout, stale state
  → You're asking "is the right data reaching the right depth?"

It's a CONDUIT problem if:
  → The output exists but is garbled, malformed, or incomplete
  → The processing worked for similar inputs before
  → You're asking "which step corrupted or blocked the data?"
  → Upstream data is fine, downstream data is wrong

It's a NETWORK problem if:
  → Multiple unrelated features break at the same time
  → The symptom is total — nothing works, not just wrong data
  → "Logging out and back in fixes everything"
  → You're asking "what do all the broken things have in common?"

Combinations

Real systems aren't purely one pattern. Your ChartPrepper app has all three, and they intersect:

The upload flow is a conduit (file → validate → extract → format → store) that terminates by entering the layer stack (store → cache → state → reactivity → screen). A bug in the conduit presents through the layers. You see a malformed summary on screen (layer 1 symptom) caused by bad LLM extraction (conduit station 3). Two different patterns, one bug.

Auth is a network that gates the conduit. If the Firebase token expires mid-upload, the LLM API call at the extraction station fails — not because extraction is broken, but because the auth network's trunk went down. The conduit looks obstructed but the lesion is in a completely different pattern.

The diagnostic move: identify the pattern first, then apply that pattern's logic. If you mix up the pattern, the diagnostic algorithm is wrong from the start — and correct execution of the wrong algorithm is still wrong.

The Surgical Analogy

An acute abdomen can involve all three patterns at once. A mesenteric artery occlusion (network — vascular territory) causes bowel ischemia (conduit — GI tract station failure) that leads to perforation and peritonitis (layer — the peritoneum is breached). You don't pick one pattern and stick with it. You triage: which pattern explains the primary pathology? The ischemia is the cause. The perforation is the complication. The peritonitis is the presentation. Treat the cause — restore flow — and the downstream patterns resolve.
